## What "semantic" actually buys you

Two pages can render pixel-identical and still feel completely different in
assistive tech. The first uses `<div>` for everything; the second uses the
right landmarks. A screen reader has no way to skip past navigation on the
first page — every paragraph reads from the top. On the second, the user
presses one key to jump to `<main>`.

That's the deal: **every semantic tag is a hint the browser, search engines,
and assistive tech can use**. Plain `<div>` says "I don't know what this is."

## The handful that matters

| Tag | Use it for |
|---|---|
| `<header>` | Page or section preamble: logo, title, intro nav. |
| `<nav>` | Primary navigation links. One main nav per page; secondary ones OK. |
| `<main>` | The unique content of the page (one per document). |
| `<article>` | Self-contained, independently distributable content. |
| `<section>` | A thematic grouping inside a document. Needs a heading. |
| `<aside>` | Tangential content — sidebar, pull quote, related links. |
| `<footer>` | Page or section conclusion: copyright, contact, related links. |

## A common refactor pattern

Before:
```html
<div class="header">
  <h1>My Blog</h1>
  <div class="nav"><a href="/">Home</a></div>
</div>
<div class="post">…</div>
<div class="footer">©</div>
```

After:
```html
<header>
  <h1>My Blog</h1>
  <nav><a href="/">Home</a></nav>
</header>
<article>…</article>
<footer>©</footer>
```

Same pixels. Wildly better accessibility, plus your CSS class names start
describing **what** instead of **where**.

## When in doubt

- Repeating layout chrome? `<header>` / `<footer>` / `<nav>`.
- A standalone unit you'd RSS-syndicate? `<article>`.
- A heading-led region of a longer document? `<section>`.
- The page's distinctive content? `<main>` (exactly once).
- Tangential stuff that won't break the page if removed? `<aside>`.
