      задача выполнена. Явный критерий — машинно-проверяемое условие: «CI

## Где агент живёт в SDLC

Жизненный цикл разработки — это последовательность шагов с конкретными
входами и выходами: триаж задач → планирование → реализация → ревью →
тестирование → деплой → эксплуатация → обратная связь. Каждый шаг
производит **артефакты** (issue, PR, отчёт, тэг релиза, инцидент), и
именно артефакты, а не действия, определяют состояние системы.

Агент в SDLC встаёт **между двумя артефактами**: он принимает один на
вход и обязан произвести другой на выход. Это даёт два важных следствия.

1. Если у шага нет входного и выходного артефакта, агент не должен
   на нём стоять — нечего проверить.
2. Артефакт — это «контракт»: PR с описанием, ссылкой на issue и
   статусом CI, а не «агент завершился без исключений».

Полезный ментальный образ: GitHub становится **системой записей и
плоскостью управления** для агентов. То, что не попало в issue/PR/check
run/workflow run, де-факто не произошло — этого нельзя ни отревьюить,
ни откатить.

<Callout type="tip" label="Правило большого пальца">
Если для шага нельзя назвать конкретный входной и выходной артефакт за
30 секунд, агенту на этом шаге пока делать нечего. Сначала зафиксируйте
контракт, потом подключайте автоматизацию.
</Callout>

## Какие шаги выполняет хороший агент

Шаг агента — это атомарное действие со структурой:
**вход → решение → действие → проверка → артефакт**.

Типичный цикл по шагам:

1. **Чтение контекста.** Issue, релевантные файлы репозитория, ранее
   созданные PR, документация, конфиги CI. Без контекста все
   последующие шаги — угадывание.
2. **Построение плана.** Список конкретных подшагов в структурированном
   виде (markdown, JSON). План — отдельный артефакт, который можно
   читать и оспаривать.
3. **Действие.** Изменение файлов, запуск команд, вызов инструментов.
   Чем уже скоуп действия, тем легче откатить.
4. **Проверка.** Прогон тестов, линт, статусы CI, дополнительные
   валидаторы. Это **машинная** сторона критерия успеха.
5. **Артефакт.** Открытие/обновление PR, комментарий с отчётом,
   обновление issue.
6. **Передача.** Уведомление человека или другого агента, если требуется
   решение или продолжение.

Все шесть пунктов оставляют следы — в issues, PR, check runs, workflow
runs. Это и есть «подотчётность по построению».

## Распространённые антишаблоны

| Антишаблон | В чём ошибка | Что делать |
|---|---|---|
| **Boil the ocean** | Один PR на десятки файлов и две задачи одновременно | Разрезать на несколько узких PR с одной целью каждый |
| **Hidden execution** | Действия без артефактов: правки в main, скрытые токены | Любое действие → артефакт в issue/PR/log |
| **Implicit success** | «Не упал» считается успехом | Явный, машинно-проверяемый критерий на каждый шаг |
| **Stateful drift** | Долгий прогон на устаревшем контексте | Перепроверка контекста между шагами; фиксация состояния в артефактах |
| **Retry-loop without escalation** | Агент крутит retry-цикл вечно | Лимит попыток + явный путь эскалации (человеку или другому агенту) |
| **Mixed-permission tool calls** | Один шаг тянет за собой high-risk действие без явного одобрения | Разделение действий по уровню риска и одобрений (см. «Уровни автономии») |

Каждый антишаблон распознаётся одной проверкой: «есть ли в этом
действии видимый артефакт и есть ли явный критерий, по которому я могу
сказать, что оно успешно?»

<Callout type="warn" label="Самая дорогая ошибка">
Retry-loop без эскалации тише всего ломается, потому что не оставляет
красных алертов: агент молча тратит токены и время, и обнаруживается
только по счёту в конце месяца или по застрявшей задаче через неделю.
Лимит попыток + явный путь эскалации — обязательный минимум для любого
автономного шага.
</Callout>

## Входы, выходы и критерии успеха

Перед тем как давать агенту задачу, опишите четыре блока — это контракт.

**Входы:**
- цель в одном предложении («исправить флейки в e2e-наборе X»);
- контекст: репозиторий, ветка, ссылки на issues/PR;
- разрешённые инструменты (что можно вызывать);
- ограничения: ветки, политики, лимиты ресурсов, дедлайны.

**Выходы:**
- основной артефакт (обычно PR или комментарий-отчёт);
- вторичные артефакты (логи, трассировки, метрики);
- состояние в трекере (issue обновлён, метка снята/проставлена).

**Критерии успеха:**
- машинно-проверяемые: CI зелёный, доля прохождения e2e ≥ M, бенчмарк
  не упал;
- человеко-проверяемые: ревьюер одобрил, scope соответствует исходному
  плану;
- что **не** считается успехом: «без исключений», «timeout вышел»,
  «закоммитил хоть что-то».

Если хотя бы один из этих блоков не определён до запуска — задачу
запускать рано. Лучше потратить пять минут на определение, чем час на
разбор того, что агент «нагенерил».

## Попробуйте

Возьмите одну реальную задачу из своего трекера, которую вы собирались
делегировать агенту. Запишите для неё четыре блока выше **до** того,
как запустить агента. Если на каком-то блоке вы зависли больше чем на
минуту — это и есть место, где у людей в команде нет общего понимания.
Сначала фиксируем это, потом зовём агента.
