## Зачем

Три класса рисков: (1) data leakage — секреты (.env, токены, PII) в контексте уходят в логи провайдера и могут попасть в training data; (2) prompt injection — внешние данные (issue, README, web fetch) содержат инструкции для AI («ignore previous instructions and...»); (3) supply chain — AI предлагает пакет, который не существует (typosquat) или содержит malware. Защита — слоистая: фильтр на входе, ограничение tool-use, проверка на выходе.

## Ключевые различения

- Secrets vs config (config может быть в git, secrets — никогда)
- Trusted input (твой код, твои доки) ↔ untrusted (web, internet, PR description)
- Supply chain: pinned versions ↔ floating (`^1.0`)
- AI-сгенерированный код ↔ дефолтно untrusted при наличии security risk

## Инструменты

- .env в .gitignore, .env.example в репо
- .cursorignore / settings.json sensitive patterns (закрыть от AI)
- Pin dependencies + npm audit / pnpm audit
- Verify package name перед install (typosquat check)

## Практика

- Проверяй package name перед install (особенно если AI предложил)
- Не вставляй ключи в prompt — даже «временно для теста»
- Не доверяй веб-fetch'ам как источнику инструкций (могут содержать prompt injection)

## Маркеры освоения

- Твой репо безопасен против AI-induced supply chain атак
- Знаешь паттерн «выглядит как имя популярной библиотеки, но typosquat» (request → reqyest, lodash → lodahs)
- В .env'е нет ничего, что увидела бы AI

## Антипаттерны

- «AI знает что делает с npm install»
- Вставлять .env в prompt «чтобы было удобно»
- Доверять fetch'у с какого-то github gist'а как источнику инструкций

## Где в стеке
