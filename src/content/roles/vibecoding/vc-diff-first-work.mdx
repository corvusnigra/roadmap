## Зачем

AI справляется с локальными изменениями (одна функция, одна фича) значительно лучше, чем с архитектурными переделками. Маленький diff = дешёвый откат, лёгкое ревью, понятный blast radius. Большая перепись = невозможно ни отревьюить, ни откатить — а если что-то сломалось в проде, поздно разбираться какая из 50 правок виновата.

## Ключевые различения

- «Маленький PR» = одна цель, ≤300 строк
- Commit-by-commit история ↔ один squash на всё
- Revert как первый план B (а не «найдём баг»)
- Feature flag для рискованных изменений

## Инструменты

- Feature branches на одну задачу
- Squash merge per задача (1 коммит в main = 1 фича)
- CI правило: PR >500 строк блокируется

## Практика

- На каждую задачу — отдельная ветка
- PR не больше 300 строк — если больше, режь
- Одна фича — один PR (не «попутно ещё и рефакторинг»)

## Маркеры освоения

- Твои PR в среднем 50-200 строк
- Знаешь, как откатить последний AI-PR за 30 секунд
- Не делаешь multi-day branches с AI

## Антипаттерны

- «Дай AI переписать всё и так посмотрим»
- Multi-week branches без merge
- Объединить рефакторинг и фичу в один PR

## Где в стеке

**Уровень 3: Дисциплина итераций.** Маленькие проверяемые итерации > большие AI-переписи. Правила, которые держат скорость и качество одновременно.
