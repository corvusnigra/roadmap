## Зачем

JOIN типы: INNER (пересечение), LEFT/RIGHT (все из левой/правой + matching из другой), FULL (всё из обеих, null на missing), CROSS (декартово произведение). WHERE — фильтрация ДО GROUP BY (на строки), HAVING — ПОСЛЕ GROUP BY (на агрегаты, например HAVING COUNT(*) > 5). Оконные функции (OVER) — агрегаты, не сворачивающие строки: ROW_NUMBER() OVER (ORDER BY x) — уникальный номер; RANK() — пропускает после ties (1,1,3); DENSE_RANK() — не пропускает (1,1,2); LAG/LEAD — предыдущая/следующая строка; SUM/AVG OVER (PARTITION BY group ORDER BY x) — running totals. CTE (WITH ...) — для читаемости и рекурсии. CTE с RECURSIVE — для деревьев.

## Ключевые различения

- INNER ↔ LEFT/RIGHT ↔ FULL ↔ CROSS JOIN
- WHERE (до GROUP BY) ↔ HAVING (после GROUP BY)
- ROW_NUMBER (1,2,3) ↔ RANK (1,1,3) ↔ DENSE_RANK (1,1,2)
- Aggregate (сворачивает) ↔ Window (сохраняет строки)

## Технология

- GROUP BY + HAVING + COUNT(*)/SUM/AVG/MIN/MAX
- ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)
- LAG / LEAD / FIRST_VALUE / LAST_VALUE
- WITH cte AS (...) SELECT ... (CTE)

## Практика (упражнения)

- Написать запрос: топ-3 товара по продажам в каждой категории (ROW_NUMBER + WHERE rn ≤ 3)
- Реализовать вторую по величине зарплату через RANK или LIMIT/OFFSET
- CROSS JOIN для генерации календаря дат

- Знаешь разницу WHERE vs HAVING без думания
- Используешь оконные функции вместо self-join, где можно
- Различаешь RANK / DENSE_RANK / ROW_NUMBER

## Антипаттерны

- HAVING вместо WHERE для не-агрегатных условий — лишний overhead
- Self-join для топ-N в группе вместо ROW_NUMBER
- Игнорировать CROSS JOIN — иногда нужен для cartesian product

## Развёрнутое объяснение

JOIN, GROUP BY/HAVING, оконные функции, CTE — основной арсенал SQL для
аналитических запросов. На middle ждут уверенного владения, без подглядывания
в документацию.

**JOIN — пять видов:**

```sql
-- INNER JOIN — только matching строки
SELECT * FROM orders o INNER JOIN users u ON o.user_id = u.id;

-- LEFT JOIN — все из левой, null если нет matching
SELECT u.name, o.id FROM users u LEFT JOIN orders o ON u.id = o.user_id;
--   ↑ у юзеров без заказов o.id будет NULL

-- RIGHT JOIN — зеркало LEFT, редко используется
SELECT * FROM orders o RIGHT JOIN users u ON u.id = o.user_id;
-- = LEFT JOIN с переставленными таблицами

-- FULL OUTER JOIN — всё из обеих
SELECT * FROM a FULL OUTER JOIN b ON a.id = b.a_id;
-- non-matching → null в соответствующих колонках

-- CROSS JOIN — Cartesian product (без ON)
SELECT * FROM colors CROSS JOIN sizes;
--   ↑ red×s, red×m, red×l, blue×s, ...
```

Визуально:

```
INNER:   A ∩ B
LEFT:    A
RIGHT:   B
FULL:    A ∪ B (с NULL'ами)
CROSS:   A × B
```

**SEMI / ANTI joins** через EXISTS / NOT EXISTS:

```sql
-- SEMI JOIN: users, у которых ЕСТЬ хотя бы один заказ
SELECT u.* FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- ANTI JOIN: users, у которых НЕТ заказов
SELECT u.* FROM users u
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- альтернатива через LEFT JOIN + IS NULL:
SELECT u.* FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;
```

Обычно `EXISTS` читаемее. Планировщик умеет оптимизировать оба.

**WHERE vs HAVING:**

```sql
SELECT customer_id, SUM(total) as total_sum
FROM orders
WHERE created_at >= '2026-01-01'   -- фильтр СТРОК до агрегации
GROUP BY customer_id
HAVING SUM(total) > 1000;          -- фильтр ГРУПП после агрегации
```

Правило: всё, что **не** агрегат — в WHERE. Агрегаты (SUM/COUNT/AVG/MIN/MAX) —
только в HAVING. WHERE дешевле (отсеивает строки до агрегации).

**Оконные функции (Window functions, OVER):**

Главная идея: **агрегаты, не сворачивающие строки**. Сохраняют каждую строку,
добавляют рядом «контекстную» агрегацию. С PG 11+ всё мощно: фреймы,
PARTITION BY, ORDER BY:

```sql
-- ROW_NUMBER, RANK, DENSE_RANK
SELECT
    name, salary,
    ROW_NUMBER() OVER (ORDER BY salary DESC) as rn,    -- 1, 2, 3, 4, 5
    RANK()       OVER (ORDER BY salary DESC) as rk,    -- 1, 1, 3, 4, 4
    DENSE_RANK() OVER (ORDER BY salary DESC) as drk    -- 1, 1, 2, 3, 3
FROM employees;
```

Разница:
- `ROW_NUMBER` — уникальный, 1 2 3 4 5.
- `RANK` — пропускает после ties: 1 1 3 4 4.
- `DENSE_RANK` — не пропускает: 1 1 2 3 3.

```sql
-- Топ-3 продавца в каждой категории
SELECT * FROM (
    SELECT
        category, name, sales,
        ROW_NUMBER() OVER (PARTITION BY category ORDER BY sales DESC) as rn
    FROM products
) sub
WHERE rn <= 3;
```

`PARTITION BY` — разбиение на группы внутри окна.

```sql
-- LAG / LEAD — соседние строки
SELECT
    date, price,
    LAG(price)  OVER (ORDER BY date) as prev_price,
    LEAD(price) OVER (ORDER BY date) as next_price,
    price - LAG(price) OVER (ORDER BY date) as daily_change
FROM stock_prices;

-- Running total
SELECT
    date, amount,
    SUM(amount) OVER (PARTITION BY user_id ORDER BY date
                      ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) as running_total
FROM transactions;

-- Moving average (7 дней)
SELECT
    date, sales,
    AVG(sales) OVER (ORDER BY date
                     ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) as moving_avg
FROM daily_sales;
```

**CTE (Common Table Expression, WITH):**

```sql
-- Базовая CTE — читаемость
WITH active_users AS (
    SELECT * FROM users WHERE last_login > NOW() - INTERVAL '30 days'
),
their_orders AS (
    SELECT au.id as user_id, o.*
    FROM active_users au
    JOIN orders o ON o.user_id = au.id
)
SELECT * FROM their_orders WHERE total > 1000;
```

**Рекурсивная CTE — для иерархий:**

```sql
-- Подразделения и все их потомки
WITH RECURSIVE subtree AS (
    SELECT id, name, parent_id, 0 as depth
    FROM departments WHERE id = $1                  -- база
    UNION ALL
    SELECT d.id, d.name, d.parent_id, s.depth + 1   -- рекурсивный шаг
    FROM departments d
    JOIN subtree s ON d.parent_id = s.id
)
SELECT * FROM subtree;
```

В PG `WITH RECURSIVE`, в MS SQL `WITH ... AS` тоже. Рекурсия делает join до
тех пор, пока новый SELECT добавляет строки. Полезно для деревьев (категории,
организационная иерархия, граф dependencies).

**Set operations:**

```sql
-- UNION (без дубликатов)
SELECT name FROM customers UNION SELECT name FROM employees;

-- UNION ALL (с дубликатами, быстрее)
SELECT id FROM a UNION ALL SELECT id FROM b;

-- INTERSECT (общие)
SELECT id FROM a INTERSECT SELECT id FROM b;

-- EXCEPT (в a, но не в b)
SELECT id FROM a EXCEPT SELECT id FROM b;
```

UNION дороже UNION ALL — он делает sort/hash для dedupe. Если знаете, что
дубликатов нет — `UNION ALL`.

## Типичный вопрос на собесе

**Вопрос 1 (Лига, Альфа, T-Bank).** «Какие виды JOIN знаешь?»

INNER, LEFT, RIGHT, FULL OUTER, CROSS. Плюс семантические SEMI/ANTI через
EXISTS/NOT EXISTS. Доводка: «когда LEFT vs RIGHT?» — RIGHT практически не
используется (можно RIGHT через LEFT с reversed order). LEFT — стандарт.

**Вопрос 2 (Лига, T-Bank).** «WHERE vs HAVING. Что выбрать в этом запросе?»

```sql
SELECT user_id, COUNT(*) as cnt FROM orders
GROUP BY user_id;
```

Если нужно отфильтровать заказы по `created_at` — WHERE. Если отфильтровать
группы по `COUNT(*) > 5` — HAVING.

```sql
SELECT user_id, COUNT(*) as cnt FROM orders
WHERE created_at > '2026-01-01'
GROUP BY user_id
HAVING COUNT(*) > 5;
```

**Вопрос 3 (Альфа, VK).** «Найди вторую по величине зарплату.»

Не очень умный способ:
```sql
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);
```

Лучше через window:
```sql
SELECT salary FROM (
    SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) as rk
    FROM employees
) WHERE rk = 2 LIMIT 1;
```

`DENSE_RANK` — потому что если две первые зарплаты равны, мы хотим «настоящую
вторую», а не третью.

**Вопрос 4 (T-Bank, Сбер).** «Найди топ-3 товара по продажам в каждой
категории.»

Каноничный window-вопрос:
```sql
SELECT category, name, sales FROM (
    SELECT
        category, name, sales,
        ROW_NUMBER() OVER (PARTITION BY category ORDER BY sales DESC) as rn
    FROM products
) sub
WHERE rn <= 3;
```

**Вопрос 5 (Озон).** «Чем отличается RANK от DENSE_RANK?»

`RANK` пропускает позиции после ties (1, 1, 3, 4). `DENSE_RANK` не пропускает
(1, 1, 2, 3). `ROW_NUMBER` всегда уникальный (1, 2, 3, 4).

## Один день из практики

Тебе пришёл тикет: «Дашборд: для каждого пользователя за последний месяц
показать: число заказов, общую сумму, средний чек, дельту к предыдущему месяцу».

Первая попытка — много subqueries:

```sql
SELECT
    u.id, u.name,
    (SELECT COUNT(*) FROM orders WHERE user_id = u.id AND created_at >= ...) as cnt,
    (SELECT SUM(total) FROM orders WHERE user_id = u.id AND created_at >= ...) as total,
    (SELECT AVG(total) FROM orders WHERE user_id = u.id AND created_at >= ...) as avg,
    (SELECT SUM(total) FROM orders WHERE user_id = u.id AND created_at >= ... AND created_at < ...) as prev_total
FROM users u;
```

4 subqueries × 100K юзеров = 400K мини-запросов. EXPLAIN ANALYZE покажет
секунды.

Чистая версия — CTE + LEFT JOIN с агрегацией:

```sql
WITH current_month AS (
    SELECT user_id,
           COUNT(*) as cnt,
           SUM(total) as total,
           AVG(total) as avg
    FROM orders
    WHERE created_at >= DATE_TRUNC('month', NOW())
    GROUP BY user_id
),
prev_month AS (
    SELECT user_id, SUM(total) as total
    FROM orders
    WHERE created_at >= DATE_TRUNC('month', NOW() - INTERVAL '1 month')
      AND created_at < DATE_TRUNC('month', NOW())
    GROUP BY user_id
)
SELECT
    u.id, u.name,
    COALESCE(c.cnt, 0) as orders_count,
    COALESCE(c.total, 0) as month_total,
    COALESCE(c.avg, 0) as avg_check,
    COALESCE(c.total, 0) - COALESCE(p.total, 0) as delta_to_prev
FROM users u
LEFT JOIN current_month c ON c.user_id = u.id
LEFT JOIN prev_month p ON p.user_id = u.id;
```

Два scan'а по orders с агрегацией. EXPLAIN покажет HashAggregate + Hash Right
Join. Быстро.

Версия с оконными функциями (одно сканирование orders):

```sql
SELECT
    u.id, u.name,
    SUM(CASE WHEN o.created_at >= DATE_TRUNC('month', NOW()) THEN 1 ELSE 0 END) as cnt,
    SUM(CASE WHEN o.created_at >= DATE_TRUNC('month', NOW()) THEN o.total ELSE 0 END) as month_total,
    SUM(CASE WHEN o.created_at >= DATE_TRUNC('month', NOW() - INTERVAL '1 month')
              AND o.created_at < DATE_TRUNC('month', NOW())
             THEN o.total ELSE 0 END) as prev_total
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE o.created_at IS NULL OR o.created_at >= DATE_TRUNC('month', NOW() - INTERVAL '1 month')
GROUP BY u.id, u.name;
```

Один JOIN, одна агрегация. На больших данных может быть быстрее (или медленнее
— зависит от планировщика). Проверять `EXPLAIN ANALYZE`.

## Типичные провалы

Кейс 1. Где баг?

```sql
SELECT *
FROM orders o
LEFT JOIN users u ON o.user_id = u.id
WHERE u.country = 'RU';
```

`WHERE u.country = 'RU'` после LEFT JOIN превращает его в INNER. Строки orders
без user'а (null user) исключаются. Чтобы оставить — фильтр в `ON`:

```sql
LEFT JOIN users u ON o.user_id = u.id AND u.country = 'RU';
```

Кейс 2. Где баг?

```sql
SELECT user_id, COUNT(*) FROM orders
GROUP BY user_id
WHERE COUNT(*) > 5;   -- syntax error
```

`WHERE` не может использовать агрегат. Правильно — `HAVING`:

```sql
SELECT user_id, COUNT(*) FROM orders
GROUP BY user_id
HAVING COUNT(*) > 5;
```

Кейс 3. Где баг (self-join вместо window)?

```sql
-- Топ-3 заказа на user'а:
SELECT o1.*
FROM orders o1
WHERE (
    SELECT COUNT(*) FROM orders o2
    WHERE o2.user_id = o1.user_id AND o2.total > o1.total
) < 3;
```

O(n²) под капотом. На 1M заказов — секунды. Window-альтернатива:

```sql
SELECT * FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total DESC) as rn
    FROM orders
) WHERE rn <= 3;
```

O(n log n).

Кейс 4. Где баг?

```sql
SELECT COUNT(DISTINCT user_id) FROM orders o
LEFT JOIN payments p ON p.order_id = o.id;
-- результат больше, чем ожидалось
```

`LEFT JOIN` на `payments` "размножил" строки (если у заказа несколько
платежей). DISTINCT user_id сделается ПОСЛЕ join. Можно либо сначала
агрегировать payments, либо в IN/EXISTS:

```sql
SELECT COUNT(DISTINCT user_id) FROM orders o
WHERE EXISTS (SELECT 1 FROM payments p WHERE p.order_id = o.id);
```

Кейс 5. Где баг?

```sql
SELECT user_id, name FROM orders;   -- name откуда?
```

`SELECT name` не от orders, но JOIN не сделан. На PG — error «column does not
exist». На некоторых SQL — fallback (если есть колонка name где-то).

Кейс 6. Где баг (NULL semantics)?

```sql
SELECT * FROM users WHERE country != 'RU';
-- юзеров с country = NULL не вернёт
```

`NULL != 'RU'` → NULL, не TRUE. Лекарство: `country IS NULL OR country != 'RU'`,
либо `COALESCE(country, '') != 'RU'`.

Кейс 7. Где баг (порядок сортировки в window)?

```sql
SELECT *, ROW_NUMBER() OVER () as rn FROM orders;   -- порядок недетерминирован
```

`OVER()` без `ORDER BY` — порядок не гарантирован между запусками. Если нужна
стабильность — `OVER (ORDER BY id)` или другой стабильный ключ.
