## Зачем

IoC (Inversion of Control) — контейнер управляет жизненным циклом, а не код. DI (Dependency Injection) — частный случай IoC, зависимости передаются извне. Три варианта в Spring: constructor (Constructor injection — рекомендуется), setter (для optional / circular), field (через @Autowired на поле — самый плохой). Constructor injection: (1) делает зависимости видимыми в сигнатуре, (2) позволяет final-поля (immutable bean), (3) ловит circular dependency на старте, не в рантайме, (4) проще тестировать (просто new Service(mock, mock)). Field injection: (1) скрывает зависимости, (2) ломает immutability, (3) не работает в plain unit-test без Spring контейнера, (4) разрешает circular cycle. Начиная с Spring 4.3 @Autowired не нужен на единственном конструкторе.

## Ключевые различения

- IoC (контейнер управляет) ↔ DI (передача зависимостей)
- Constructor injection (best) ↔ Setter (optional) ↔ Field (worst)
- Singleton (default scope) ↔ Prototype
- @Autowired ↔ @Inject (JSR-330) ↔ @Resource (JSR-250)

## Технология

- @Component / @Service / @Repository / @Controller / @RestController
- @Configuration + @Bean — manual bean definition
- @Autowired + @Qualifier + @Primary
- ApplicationContext.getBean() (для тестов и edge cases)

## Практика (упражнения)

- Переписать field injection класс на constructor injection — увидеть, что тесты проще
- Создать circular dependency через field, потом через constructor — увидеть failure mode
- Реализовать @Bean в @Configuration с условиями (@ConditionalOnProperty)

## Маркеры освоения

- Используешь только constructor injection в новом коде
- Можешь объяснить, почему @Autowired на поле — антипаттерн
- Знаешь разницу @Autowired / @Inject / @Resource

## Антипаттерны

- @Autowired на private field — невозможно протестировать без reflection или Spring
- Circular dependency через setter / field — Spring «починит» молча
- ApplicationContext.getBean() в обычном бизнес-коде

## Развёрнутое объяснение

IoC (Inversion of Control) — паттерн: контейнер управляет жизненным циклом
объектов и их зависимостями, а не сам объект. DI (Dependency Injection) —
частный случай IoC: зависимости приходят извне (не `new` внутри класса).

Без IoC:
```java
public class OrderService {
    private final UserRepository repo = new UserRepository(new JdbcTemplate(new HikariDataSource()));
    // OrderService должен знать, как собрать всю цепочку
}
```

С IoC:
```java
@Service
public class OrderService {
    private final UserRepository repo;
    public OrderService(UserRepository repo) { this.repo = repo; }
}
// Spring собирает граф зависимостей сам
```

**Три типа DI в Spring:**

```java
// 1. Constructor injection — РЕКОМЕНДОВАННЫЙ
@Service
public class OrderService {
    private final UserRepository repo;
    private final PaymentClient client;
    public OrderService(UserRepository repo, PaymentClient client) {  // @Autowired не нужен с 4.3+
        this.repo = repo;
        this.client = client;
    }
}
// Плюсы:
// - final поля (immutability)
// - зависимости явно видны в сигнатуре
// - циркуляры ловятся на старте (Spring fail-fast)
// - тестируется БЕЗ Spring (new OrderService(mockRepo, mockClient))

// 2. Setter injection — для optional-зависимостей
@Service
public class OrderService {
    private UserRepository repo;
    @Autowired(required = false)
    public void setRepo(UserRepository repo) { this.repo = repo; }
}
// Применение: redko, обычно для legacy-кода

// 3. Field injection — АНТИПАТТЕРН
@Service
public class OrderService {
    @Autowired private UserRepository repo;   // ✗
}
// Минусы:
// - нельзя final
// - зависимости скрыты, можно вообразить «без репозитория не работает» по signature
// - тестируется только с @SpringBootTest или ReflectionTestUtils.setField
// - разрешает circular dependency (Spring «починит» сам, скрывая баг)
// - IntelliJ ругается warning'ом
```

**Lombok делает constructor injection ещё короче:**

```java
@Service
@RequiredArgsConstructor   // генерирует конструктор для всех final-полей
public class OrderService {
    private final UserRepository repo;
    private final PaymentClient client;
    // нет boilerplate
}
```

**Аннотации `@Component`-семейства.** Все они эквивалентны как DI-указатели,
различаются только семантически и для AOP:
- `@Component` — generic component.
- `@Service` — бизнес-слой (для IDE-навигации, не более).
- `@Repository` — DAO. Spring добавляет `PersistenceExceptionTranslationPostProcessor`,
  который оборачивает JDBC-исключения в Spring `DataAccessException`.
- `@Controller` / `@RestController` — web-слой.

Все 4 — это `@Component` под капотом + дополнительная семантика.

**`@Configuration` + `@Bean`** для случаев, когда:
- Третий пакетный класс (не ваш) — нет возможности добавить `@Component`.
- Нужно сложное создание (factory pattern, условная конфигурация).
- Конфигурация бинов внешней библиотеки (HikariDataSource, Jackson).

```java
@Configuration
public class DataSourceConfig {
    @Bean
    @ConfigurationProperties("app.datasource")
    public DataSource dataSource() {
        return new HikariDataSource();
    }

    @Bean
    @ConditionalOnProperty(name = "cache.enabled", havingValue = "true")
    public CacheManager cacheManager() {
        return new ConcurrentMapCacheManager();
    }
}
```

**Qualifier и Primary** — разрешение неоднозначности:

```java
public interface Notifier { void send(String msg); }

@Component("email") class EmailNotifier implements Notifier { ... }
@Component("sms")   class SmsNotifier   implements Notifier { ... }
@Component @Primary class PrimaryNotifier implements Notifier { ... }

@Service
public class AlertService {
    public AlertService(@Qualifier("sms") Notifier notifier) { ... }   // явный выбор
    // или
    public AlertService(Notifier notifier) { ... }                       // @Primary winner
}
```

**Collection injection** — все бины интерфейса:

```java
@Service
public class MultiNotifier {
    private final List<Notifier> all;
    public MultiNotifier(List<Notifier> all) { this.all = all; }  // получит ВСЕ Notifier'ы
    public void broadcast(String msg) { all.forEach(n -> n.send(msg)); }
}

// Map<String, Notifier> — bean name → bean
public MultiNotifier(Map<String, Notifier> byName) { ... }
```

Полезно для strategy pattern, plugin-архитектур.

**Circular dependency.** Граф зависимостей с циклом — A → B → A. С
constructor-only — Spring **отказывается** запускать (`BeanCurrentlyInCreationException`).
С setter/field — Spring «исхитряется» через early reference (отдаёт ссылку до
завершения init), и приложение стартует, но **бин может быть в неконсистентном
состоянии**. С Spring Boot 2.6+ — циклические зависимости запрещены по
дефолту (нужно `spring.main.allow-circular-references=true`, но это считается
антипаттерном).

Решение цикла:
1. **Reorganize**: вынести общую часть в третий компонент.
2. **`@Lazy`**: один из бинов получает прокси, реальное создание откладывается
   до первого использования (`@Lazy` в constructor argument).
3. **`ObjectProvider<T>`** или `Provider<T>` (JSR-330) — отложенное получение.

**JSR-330 (`javax.inject` / `jakarta.inject`).** Стандартный портативный API:

```java
@Named                          // = @Component
public class OrderService {
    @Inject UserRepository repo; // = @Autowired
}
```

Spring поддерживает, но в реальном проекте чаще @Autowired/@Component из-за
доступа к расширенным фичам Spring (qualifiers, @Lazy).

## Типичный вопрос на собесе

**Вопрос 1 (Лига, T-Bank, Альфа, Сбер, VK).** «Что такое IoC и DI?»

IoC — «обратное управление»: контейнер управляет жизненным циклом объектов,
а не сам код. DI — частный случай: зависимости приходят извне (через
конструктор/setter/поле).

Доводка: «Что Spring управляет?» — создание, scope (singleton/prototype),
init/destroy, autowire зависимостей, прокси для AOP, dependency graph
validation.

**Вопрос 2 (Лига, Альфа, T-Bank, VK).** «Constructor vs setter vs field
injection. Что выбрать?»

Constructor: рекомендованный default. Final поля, явные зависимости, тестируется
без Spring, циркуляры ловятся на старте.

Setter: для optional-зависимостей (`@Autowired(required = false)`) или
конфигуряции в legacy-коде.

Field: антипаттерн. Скрывает зависимости, ломает immutability, не работает в
plain unit тестах, разрешает циркуляры без сигнала.

**Вопрос 3 (T-Bank, Сбер).** «Что произойдёт при circular dependency?»

С constructor-only: `BeanCurrentlyInCreationException`, Spring отказывается
стартовать. С setter/field (если разрешено): Spring «исхитряется» — отдаёт
ссылку до init, но бин может быть partially-constructed. С Spring Boot 2.6+
циклы запрещены по дефолту.

Решения: reorganize в третий компонент, `@Lazy`, `ObjectProvider`.

**Вопрос 4 (Альфа).** «Чем `@Service` отличается от `@Component`?»

Семантически — ничем. Технически — `@Service`, `@Repository`, `@Controller` —
это `@Component` + дополнительная роль. У `@Repository` есть бонус:
`PersistenceExceptionTranslationPostProcessor` оборачивает SQLException в
DataAccessException (унифицированная иерархия).

**Вопрос 5 (Озон, VK).** «Можно ли инжектить `List<T>` всех реализаций? Какой
порядок?»

Да: `private final List<Notifier> all;` — Spring соберёт все бины Notifier.
Порядок:
- По `@Order` или `Ordered` интерфейсу.
- Если не указано — по имени бина или по порядку scan'а (не гарантирован).

Также есть `Map<String, T>` — `bean name → bean`.

## Один день из практики

Тебе пришёл legacy-проект на migration. В нём:

```java
@Service
public class UserService {
    @Autowired UserRepository repo;
    @Autowired EmailSender emailer;
    @Autowired @Lazy NotificationService notif;

    public void createUser(...) {
        repo.save(...);
        emailer.send(...);
        notif.notify(...);
    }
}

@Service
public class NotificationService {
    @Autowired UserService userService;   // circular!
}
```

Шаг 1. Проблемы:
- field injection везде.
- Циркуляр UserService ↔ NotificationService, разрешён через `@Lazy`.
- ImageProcessor (внутри) — singleton, но не thread-safe.

Шаг 2. Рефакторим UserService на constructor injection:

```java
@Service
@RequiredArgsConstructor
public class UserService {
    private final UserRepository repo;
    private final EmailSender emailer;
    private final ApplicationEventPublisher events;   // вместо NotificationService

    public void createUser(...) {
        User u = repo.save(...);
        emailer.send(...);
        events.publishEvent(new UserCreatedEvent(u));   // развязали через events
    }
}

@Service
@RequiredArgsConstructor
public class NotificationService {
    private final UserRepository userRepo;   // прямая зависимость без UserService

    @EventListener
    public void onUserCreated(UserCreatedEvent e) {
        // отправить уведомление
    }
}
```

Циркуляр развязан через ApplicationEvent. UserService больше не знает о
NotificationService. Каждый — самостоятельный.

Шаг 3. Тесты теперь работают без Spring:

```java
class UserServiceTest {
    UserRepository repo = mock(UserRepository.class);
    EmailSender emailer = mock(EmailSender.class);
    ApplicationEventPublisher events = mock(ApplicationEventPublisher.class);
    UserService service = new UserService(repo, emailer, events);
    // вообще без @SpringBootTest, в 100мс
}
```

Старый код с field injection требовал `@SpringBootTest` (full context, 5+
секунд на тест).

## Типичные провалы

Кейс 1. Где баг?

```java
@Service
public class Cache {
    @Autowired
    private static Repository repo;   // ✗ static!
}
```

Spring не делает DI в static-поля. `repo` будет null. Лекарство:
constructor injection (через нестатический setter, который копирует в static —
если очень нужно).

Кейс 2. Где баг?

```java
@Service
public class Service {
    @Autowired
    private final Dep dep;   // final + field injection
}
```

`final` поле не может быть присвоено DI (field injection без `@RequiredArgsConstructor`
или явного конструктора). Компилируется с warning'ом, но это не то, что
кажется. Лекарство: constructor.

Кейс 3. Где баг?

```java
public Service(Dep1 d1, Dep2 d2, Dep3 d3, Dep4 d4, Dep5 d5, Dep6 d6) { ... }
```

6+ зависимостей в одном классе — признак violation Single Responsibility. Класс
делает слишком много. Лекарство: разбить на 2-3 более узких сервиса.

Кейс 4. Где баг (qualifier collision)?

```java
@Component("notifier")
class EmailNotifier { ... }

@Component("notifier")
class SmsNotifier { ... }
// `BeanDefinitionOverrideException` (с Spring Boot 2.1+)
```

Один и тот же bean name — конфликт. До Spring Boot 2.1 один молча перезаписывал
другой. Сейчас — error. Лекарство: разные имена или `@Primary`.

Кейс 5. Где баг (interface vs class)?

```java
public interface Repository { ... }

@Component
public class JpaRepository implements Repository { ... }

@Service
@RequiredArgsConstructor
public class Service {
    private final JpaRepository repo;   // injection по конкретному классу
}
```

Не plug-and-play — Service не может работать с другой реализацией Repository.
Лучше: `private final Repository repo;` — DI по интерфейсу.

Кейс 6. Где баг (ApplicationContext)?

```java
@Service
public class Service {
    @Autowired ApplicationContext context;

    public void doWork(String type) {
        Strategy s = context.getBean(type, Strategy.class);   // service locator
        s.execute();
    }
}
```

Service locator anti-pattern. Тесты сложные, coupling с Spring. Лекарство:
`Map<String, Strategy>` injection.

Кейс 7. Где баг (eager vs lazy)?

```java
@Service
@RequiredArgsConstructor
public class Slow {
    private final ExpensiveResource resource;   // создаётся eagerly даже если Slow не используется
}
```

Spring eagerly создаёт все singleton-бины на старте. Если бин редкий, но
тяжёлый — старт долгий. Лекарство: `@Lazy` на классе (создание по первому
использованию) или на конкретной зависимости.
