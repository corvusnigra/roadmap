## Зачем

N+1: 1 запрос для main collection + N запросов для каждой lazy-relation. List<Order> orders = repository.findAll(); for (Order o : orders) o.getItems().size(); — 1 + N SELECT. Решения по сложности: (1) JOIN FETCH в JPQL: SELECT o FROM Order o JOIN FETCH o.items — 1 запрос с JOIN, но дубликаты строк (нужен DISTINCT или Set), не работает с пагинацией (HHH000104 warning); (2) @EntityGraph("order-with-items") — декларативное FETCH; (3) @BatchSize(size = 100) — IN-batching, 1 + N/100 запросов; (4) DTO projection — самое чистое, query напрямую в DTO без JPA-сущности; (5) Hibernate fetch profiles. EAGER — НЕ решение N+1, она просто меняет lazy на always-load, что хуже в большинстве сценариев.

## Ключевые различения

- N+1 (lazy без JOIN FETCH) ↔ Cartesian explosion (несколько JOIN FETCH в одном)
- JOIN FETCH ↔ @EntityGraph ↔ @BatchSize ↔ DTO projection
- EAGER — НЕ решение N+1 (просто всегда грузит)
- DISTINCT при JOIN FETCH (default ON для коллекций)

## Технология

- JPQL: SELECT o FROM Order o JOIN FETCH o.items
- @EntityGraph(attributePaths = {"items"})
- @BatchSize(size = 100) на @OneToMany
- DTO projection: SELECT new com.x.OrderDto(o.id, o.total) FROM Order o

## Практика (упражнения)

- Включить spring.jpa.show-sql + p6spy — увидеть N+1 в логе
- Решить через JOIN FETCH, заметить дубликаты — добавить DISTINCT
- Заменить на @EntityGraph — увидеть тот же SQL

## Развёрнутое объяснение

N+1 — самая частая SQL-проблема в JPA/Hibernate-приложениях. Спросят в 8/8
компаний. На middle ждут понимания **всех 4 решений** и **когда какое
применять**.

**Что такое N+1:**

```java
@Entity
public class Order {
    @Id Long id;
    @ManyToOne(fetch = FetchType.LAZY) User user;
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY) List<OrderItem> items;
}

// Код:
List<Order> orders = orderRepo.findAll();         // 1 запрос: SELECT * FROM orders
for (Order o : orders) {
    System.out.println(o.getItems().size());      // N запросов: один на каждый order
}
```

Логи Hibernate:

```sql
SELECT * FROM orders LIMIT 100;           -- 1 запрос
SELECT * FROM order_items WHERE order_id = 1;
SELECT * FROM order_items WHERE order_id = 2;
SELECT * FROM order_items WHERE order_id = 3;
...
SELECT * FROM order_items WHERE order_id = 100;   -- N запросов
```

На 100 orders — 101 SQL-запрос. На 10K orders — 10001. Latency растёт линейно.
На уровне приложения это выглядит так: «загрузка orders заняла 30мс», а
total — 3 секунды.

**Откуда берётся.** Hibernate lazy-loading работает через прокси. При обращении
к `getItems()` или `getUser()` — Hibernate делает SELECT по foreign key. Каждый
доступ к lazy-полю — отдельный запрос, если поле не было предзагружено.

**4 решения по сложности:**

### 1. JOIN FETCH в JPQL

```java
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.status = :status")
List<Order> findByStatusWithItems(@Param("status") OrderStatus status);
```

Hibernate генерирует один SQL с LEFT JOIN. Items приходят в той же query.

**Проблемы JOIN FETCH:**

1. **Декартово произведение** для `@OneToMany`:

   ```sql
   SELECT o.*, i.* FROM orders o LEFT JOIN order_items i ON i.order_id = o.id;
   -- Если у order'а 5 items, в результате — 5 строк с дубликатами order-полей.
   ```

   Hibernate dedupe'ит на уровне entities, но если возвращаемый список не Set, в
   результате duplicates. Лекарство: `SELECT DISTINCT o FROM Order o JOIN FETCH
   o.items`.

2. **Cartesian explosion** при двух OneToMany:

   ```java
   @Query("SELECT o FROM Order o JOIN FETCH o.items JOIN FETCH o.payments")
   ```

   На 10 items × 5 payments = 50 строк per order. На большом datasete OOM.
   Hibernate с 6.x бросает `MultipleBagFetchException` для двух bag.

3. **Пагинация не работает** с JOIN FETCH:

   ```java
   Page<Order> findAll(Pageable p);   // с join fetch
   ```

   Hibernate выдаст warning `HHH000104: firstResult/maxResults specified with
   collection fetch; applying in memory!`. Загрузит **весь** result set, потом
   пагинирует в Java. На 1M строк → OOM.

### 2. `@EntityGraph`

```java
@EntityGraph(attributePaths = {"items", "user"})
@Query("SELECT o FROM Order o WHERE o.status = :status")
List<Order> findByStatusWithItems(@Param("status") OrderStatus status);

// или объявленный граф:
@NamedEntityGraph(name = "Order.withItems", attributeNodes = @NamedAttributeNode("items"))
@Entity
public class Order { ... }

@EntityGraph("Order.withItems")
Optional<Order> findById(Long id);
```

Декларативный аналог JOIN FETCH. Под капотом — тот же SQL. Преимущество:
переиспользуемые graph'ы, понятнее в коде.

### 3. `@BatchSize` — IN-batching

```java
@OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
@BatchSize(size = 100)
List<OrderItem> items;
```

При первом обращении к items одного order'а Hibernate подмечает остальные
unfetched orders (до 100) и делает:

```sql
SELECT * FROM order_items WHERE order_id IN (1, 2, 3, ..., 100);
```

Вместо N запросов — N/batch_size. На 1000 orders + batch 100 = 10 запросов.

Преимущества:
- Не нужно менять repository методы.
- Работает с пагинацией.
- Глобально настраивается в `hibernate.default_batch_fetch_size=100`.

Недостатки:
- Всё равно второй round-trip к БД.
- Меньше контроля над тем, что запрашивается.

### 4. DTO projection

```java
public record OrderSummary(Long id, BigDecimal total, String userName) {}

@Query("""
    SELECT new com.acme.OrderSummary(o.id, o.total, u.name)
    FROM Order o JOIN o.user u
    WHERE o.status = :status
""")
List<OrderSummary> findSummariesByStatus(@Param("status") OrderStatus status);
```

Запрос **напрямую в DTO**, без поднятия entity. Самое быстрое и memory-friendly:
- Один SQL.
- Только нужные колонки (SELECT id, total, name — не весь Order/User).
- Нет Hibernate entity overhead (нет proxy, нет dirty checking, не сохраняется
  в PersistenceContext).

Минус: read-only, нельзя «updateить и save».

**EAGER — не решение N+1.**

```java
@OneToMany(fetch = FetchType.EAGER)   // ✗
```

EAGER загружает items ВСЕГДА, даже когда они не нужны. Худшее из обоих миров:
если нужны — N+1 не решён (Hibernate всё равно может сделать дополнительные
запросы для других lazy). Если не нужны — лишний overhead. **Никогда не
ставить EAGER** на коллекциях. Maybe на критичных @ManyToOne (User в Order),
но и там лучше JOIN FETCH в нужных queries.

**Когда какое решение:**

```
JOIN FETCH         — для конкретных query, где нужно forward'ить отношения.
                     Простой single-association + ToOne. NO Pageable.

@EntityGraph       — те же случаи, но декларативно. Многоразовые graph'ы.

@BatchSize         — глобально для всех методов, в которых уже встречается N+1.
                     С пагинацией. На лёгких ассоциациях.

DTO projection     — для read-heavy endpoint'ов (списки, dashboards).
                     Максимальная performance.

EAGER              — почти никогда.
```

## Типичный вопрос на собесе

**Вопрос 1 (Лига, Альфа, T-Bank, Яндекс, Сбер, VK, Озон, СберDP).** «Что такое
N+1 и как с ним бороться?»

N+1: 1 запрос для main collection + N запросов для каждого lazy relation.
Решения: JOIN FETCH (для single ToMany), @EntityGraph (декларативно),
@BatchSize (IN-batching), DTO projection (для read-only).

EAGER — не решение, это другая проблема.

**Вопрос 2 (Лига).** «JOIN FETCH с пагинацией — что не так?»

Hibernate не может корректно пагинировать после JOIN с One-To-Many: SQL
LIMIT отсекает по строкам, не по entity. Hibernate выдаёт warning
`HHH000104` и **загружает весь result set в память**, потом пагинирует.
OOM на больших таблицах.

Лекарство: либо `@BatchSize` (пагинация работает), либо двухэтапный fetch
(сначала id'ы с пагинацией, потом `WHERE id IN (...)` с JOIN FETCH).

**Вопрос 3 (T-Bank).** «Что не так?»

```java
@OneToMany(fetch = FetchType.EAGER)
List<OrderItem> items;
```

EAGER загружает items всегда. Если на странице 100 orders × 50 items, это
5000 строк прямо при запросе списка. На страничном endpoint — катастрофа.
Лекарство: LAZY + JOIN FETCH/EntityGraph только там, где нужно.

**Вопрос 4 (Альфа).** «Как увидеть N+1 в логах?»

`spring.jpa.show-sql=true` + `spring.jpa.properties.hibernate.format_sql=true`.
Лучше — `p6spy` (proxy datasource, форматирует SQL, считает duration). Для
production-prevention — Hibernate Statistics + sentry, или
**spring-boot-starter-jpa** с `spring.jpa.properties.hibernate.generate_statistics=true`.

## Один день из практики

Тебя позвали: «endpoint `/orders` иногда отдаёт за 30 секунд». Включаешь
p6spy и видишь:

```
SELECT * FROM orders LIMIT 100;                    -- 1ms
SELECT * FROM users WHERE id = 1;                  -- 0.5ms
SELECT * FROM users WHERE id = 2;                  -- 0.5ms
SELECT * FROM users WHERE id = 3;                  -- 0.5ms
... (100 раз)
SELECT * FROM order_items WHERE order_id = 1;      -- 0.5ms
SELECT * FROM order_items WHERE order_id = 2;      -- 0.5ms
... (100 раз)
```

Сразу 200 N+1 запросов + 1 main. На latent сети 1ms × 201 = 200ms.

Открываем сервис:

```java
@Service
public class OrderService {
    private final OrderRepository repo;

    public List<OrderDto> findRecent() {
        return repo.findTop100ByOrderByCreatedAtDesc().stream()
            .map(o -> new OrderDto(
                o.getId(),
                o.getUser().getName(),                       // N+1!
                o.getItems().stream()                        // N+1!
                    .map(i -> new OrderItemDto(i.getName()))
                    .toList()
            ))
            .toList();
    }
}
```

Решение 1 — DTO projection через JPQL:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {
    @Query("""
        SELECT new com.acme.OrderDto(o.id, u.name)
        FROM Order o JOIN o.user u
        ORDER BY o.createdAt DESC
    """)
    List<OrderDto> findRecentSummaries(Pageable pageable);
}
```

Но если в DTO нужны items, projection становится сложной. Альтернатива —
двухэтапный fetch:

```java
public List<OrderDto> findRecent() {
    // Шаг 1: только orders с user (через JOIN FETCH), без items
    List<Order> orders = repo.findTop100WithUser();

    // Шаг 2: батчем подтянуть items
    List<Long> orderIds = orders.stream().map(Order::getId).toList();
    Map<Long, List<OrderItem>> itemsByOrder = itemRepo.findByOrderIdIn(orderIds).stream()
        .collect(Collectors.groupingBy(i -> i.getOrder().getId()));

    return orders.stream()
        .map(o -> new OrderDto(
            o.getId(),
            o.getUser().getName(),
            itemsByOrder.getOrDefault(o.getId(), List.of()).stream()
                .map(i -> new OrderItemDto(i.getName()))
                .toList()
        ))
        .toList();
}

@Query("SELECT o FROM Order o JOIN FETCH o.user ORDER BY o.createdAt DESC")
List<Order> findTop100WithUser(Pageable p);
```

2 запроса вместо 201. 30 сек → 50мс.

Решение 2 — `@BatchSize` (минимальный диф):

```java
@Entity
public class Order {
    @ManyToOne(fetch = FetchType.LAZY)
    @BatchSize(size = 100)
    User user;

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    @BatchSize(size = 100)
    List<OrderItem> items;
}
```

После этого Hibernate группирует запросы в IN. Логи:

```
SELECT * FROM orders LIMIT 100;
SELECT * FROM users WHERE id IN (1, 2, ..., 100);
SELECT * FROM order_items WHERE order_id IN (1, 2, ..., 100);
```

3 запроса вместо 201. Простое изменение, без переписывания.

## Типичные провалы

Кейс 1. Где баг?

```java
@Query("SELECT o FROM Order o JOIN FETCH o.items JOIN FETCH o.payments")
List<Order> findAllWith();
```

Cartesian explosion. 1M строк в результате при 100 orders × 10 items × 10
payments. Hibernate 6 бросит `MultipleBagFetchException`. Лекарство:
двухэтапный fetch, или одно отношение через FETCH + второе через @BatchSize.

Кейс 2. Где баг?

```java
@Query("SELECT o FROM Order o JOIN FETCH o.items")
Page<Order> findAll(Pageable p);
```

Pagination + JOIN FETCH на коллекцию = HHH000104 warning + in-memory pagination.
Hibernate загрузит ВСЁ. Лекарство: pagination на entity id, потом
`WHERE id IN (...)` с FETCH:

```java
@Query("SELECT o.id FROM Order o ORDER BY o.createdAt")
Page<Long> findIds(Pageable p);

@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id IN :ids")
List<Order> findWithItems(@Param("ids") List<Long> ids);
```

Кейс 3. Где баг?

```java
@OneToMany(fetch = FetchType.EAGER)
List<OrderItem> items;
```

EAGER на коллекциях — антипаттерн. Каждый load Order грузит items, даже если
не нужны. Лекарство: LAZY + явный fetch там, где нужно.

Кейс 4. Где баг?

```java
@Transactional(readOnly = true)
public List<OrderDto> findRecent() {
    return repo.findTop100ByOrderByCreatedAtDesc().stream()
        .map(o -> new OrderDto(o.getId(), o.getUser().getName()))
        .toList();
}
// БЕЗ JOIN FETCH — N+1
```

В readOnly TX это всё равно N+1. Лекарство: либо @EntityGraph, либо DTO
projection в repository.

Кейс 5. Где баг?

```java
@Service
public class OrderService {
    @Transactional(readOnly = true)
    public Order findById(Long id) {
        return repo.findById(id).orElseThrow();
    }
}

@RestController
public class OrderController {
    @GetMapping("/orders/{id}")
    public OrderDto get(@PathVariable Long id) {
        Order o = orderService.findById(id);
        return new OrderDto(o.getId(),
                            o.getItems().stream()...   // LazyInitializationException
                                                          // (если spring.jpa.open-in-view=false)
        );
    }
}
```

Lazy collection доступ ВНЕ transaction → `LazyInitializationException`. По
дефолту в Spring Boot `open-in-view=true` (антипаттерн — держит TX на всё
время response). Лекарство: `open-in-view=false` + DTO внутри service.

Кейс 6. Где баг (1+1+1)?

```java
@OneToMany(mappedBy = "order")
List<OrderItem> items;

@OneToMany(mappedBy = "order")
List<Payment> payments;

@ManyToOne
User user;

// Запрос: 1 + 1(user) + N(items) + N(payments) — 1 + 1 + 2N
```

Лекарство: либо два EntityGraph в один call (по-разному в Hibernate), либо
двухэтапный fetch (id'ы → batch IN), либо BatchSize на всех.

Кейс 7. Где баг (Spring Data Specifications)?

```java
Specification<Order> spec = (root, query, cb) -> cb.equal(root.get("status"), "NEW");
repo.findAll(spec);
// внутри map к DTO — N+1
```

`findAll(Spec)` не делает FETCH. Лекарство: внутри Specification
`root.fetch("items", JoinType.LEFT)` либо `EntityGraph` через `repo.findAll(spec, Pageable, EntityGraph<Order>)` (в Spring Data 2.x+).
