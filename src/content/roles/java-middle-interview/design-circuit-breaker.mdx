## Зачем

Circuit Breaker — паттерн от каскадного отказа. Три состояния: CLOSED (всё OK, считаем failure rate), OPEN (failure rate > threshold — все запросы fail-fast без вызова downstream), HALF_OPEN (после timeout — пропускаем пробные запросы; success → CLOSED, fail → OPEN). Без CB одно медленное API забивает thread pool и роняет весь сервис. Retry — повтор при failure, но: (1) опасен retry storm — все клиенты одновременно ретраят после обрыва, downstream добивает; решение — exponential backoff + jitter (random шум для расфазировки). (2) Retry имеет смысл только для transient errors (network, timeout), не для 4xx. Resilience4j — composable: Retry + CircuitBreaker + RateLimiter + Bulkhead + TimeLimiter, каждый wrapper над Supplier/Function.

## Ключевые различения

- CLOSED (норма) ↔ OPEN (fail-fast) ↔ HALF_OPEN (пробные запросы)
- Retry без jitter ↔ Retry с exponential backoff + jitter
- Transient errors (network, timeout) ↔ Permanent (4xx, validation)
- Hystrix (legacy) ↔ Resilience4j (modern)

## Технология

- Resilience4j: CircuitBreaker, Retry, RateLimiter, Bulkhead, TimeLimiter
- Spring Cloud Circuit Breaker (abstraction)
- @CircuitBreaker / @Retry аннотации
- Micrometer интеграция для метрик

## Практика (упражнения)

- Обернуть REST-клиент в @CircuitBreaker с failureRateThreshold = 50%
- Спровоцировать OPEN через 5 failures, дождаться HALF_OPEN
- Добавить retry с exponential backoff (initial 100ms, multiplier 2, jitter 0.5)

## Маркеры освоения

- Используешь CircuitBreaker для всех внешних HTTP
- Не делаешь retry на 4xx
- Знаешь, зачем jitter в retry

## Антипаттерны

- Retry без jitter — retry storm на восстанавливающийся сервис
- Бесконечный retry без max attempts — заваливаешь pool
- Retry на POST без idempotency — двойная операция

## Развёрнутое объяснение

Circuit Breaker — паттерн отказоустойчивости, защищающий сервис от
каскадных отказов при сбое downstream'а. Имя — от электрических предохранителей:
при перегрузке цепь размыкается.

**Три состояния:**

```
       failure rate > threshold
   ┌─────────────────────────┐
   │                          ▼
CLOSED ────────────────────► OPEN
   ▲                          │
   │                          │ waitDurationInOpenState (e.g. 30s)
   │                          ▼
   │  все calls           HALF_OPEN
   │  successful           │  │
   │  in HALF_OPEN         │  │  permittedCallsInHalfOpen
   └──────────────────────┘  │  если хоть один fail
                              ▼
                            OPEN
```

```
CLOSED:    нормальная работа. CB считает success/failure rate.
           На failure rate > threshold (default 50%) → OPEN.

OPEN:      все вызовы fail-fast (CallNotPermittedException).
           Downstream не дёргаем — даём ему восстановиться.
           Через waitDurationInOpenState (например, 30 сек) → HALF_OPEN.

HALF_OPEN: пробуем N запросов (permittedNumberOfCallsInHalfOpenState, default 10).
           Если все успешны → CLOSED.
           Если хотя бы один fail → OPEN.
```

**Зачем нужен Circuit Breaker:**

Без CB: внешний сервис тормозит (timeout 30 сек на каждый запрос). Наш сервис
получает 100 RPS. Через 30 сек у нас 3000 threads висят в ожидании. Tomcat
pool (200) исчерпан. Все остальные endpoint'ы виснут. **Каскадный отказ**.

С CB: после 5 fails — CB OPEN. Все следующие запросы фейлятся через 1ms (без
обращения к downstream'у). Наш сервис продолжает работать, отдаёт fallback
(cached данные или 503). Downstream восстановился — CB перешёл в HALF_OPEN,
пропустил несколько запросов, и если они успешны → CLOSED, нормальная работа.

**Resilience4j (наследник Hystrix):**

```java
@Service
public class PaymentService {
    @CircuitBreaker(name = "payment", fallbackMethod = "fallback")
    @Retry(name = "payment")
    @TimeLimiter(name = "payment")
    public CompletableFuture<PaymentResult> charge(PaymentRequest req) {
        return CompletableFuture.supplyAsync(() -> client.charge(req));
    }

    public CompletableFuture<PaymentResult> fallback(PaymentRequest req, Throwable t) {
        log.warn("Payment circuit open", t);
        return CompletableFuture.completedFuture(PaymentResult.declined("temporary unavailable"));
    }
}
```

```yaml
resilience4j:
  circuitbreaker:
    instances:
      payment:
        slidingWindowType: COUNT_BASED          # или TIME_BASED
        slidingWindowSize: 100
        failureRateThreshold: 50                 # %
        slowCallRateThreshold: 80
        slowCallDurationThreshold: 2s
        waitDurationInOpenState: 30s
        permittedNumberOfCallsInHalfOpenState: 10
        minimumNumberOfCalls: 20                 # не открываемся раньше 20 calls
        automaticTransitionFromOpenToHalfOpenEnabled: true
  retry:
    instances:
      payment:
        maxAttempts: 3
        waitDuration: 100ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
        retryExceptions:
          - java.net.ConnectException
          - java.util.concurrent.TimeoutException
        ignoreExceptions:
          - com.acme.BusinessException
  timelimiter:
    instances:
      payment:
        timeoutDuration: 2s
        cancelRunningFuture: true
```

**Retry — обязательный compañero:**

```java
// Naive retry без backoff:
for (int i = 0; i < 3; i++) {
    try { return client.call(); }
    catch (Exception e) { if (i == 2) throw e; }
}
// При обрыве downstream'а — 3 запроса подряд за 100мс.
// Если 1000 клиентов делают так же — 3000 запросов на восстанавливающийся.
// Retry storm. Downstream снова падает.

// С exponential backoff + jitter:
long delay = 100;
for (int i = 0; i < 3; i++) {
    try { return client.call(); }
    catch (Exception e) {
        if (i == 2) throw e;
        long jitter = ThreadLocalRandom.current().nextLong(delay / 2);
        Thread.sleep(delay + jitter);
        delay *= 2;
    }
}
// Retry на 100±50ms, 200±100ms, 400±200ms.
// Jitter рассинхронизирует клиентов, downstream получает плавную нагрузку.
```

**Когда retry имеет смысл:**

```
Retry OK:
  - ConnectException, TimeoutException — transient network.
  - 503 Service Unavailable.
  - 504 Gateway Timeout.
  - 429 Too Many Requests (с Retry-After header).

Retry НЕ OK:
  - 4xx (кроме 408, 429) — клиентская ошибка, повтор не поможет.
  - BusinessException — логическая ошибка.
  - На POST без idempotency-key — двойное списание!
```

**Bulkhead — изоляция ресурсов:**

```java
@Bulkhead(name = "payment", type = Bulkhead.Type.SEMAPHORE)
public PaymentResult charge() { ... }

// SEMAPHORE — ограничивает число concurrent calls.
// THREADPOOL — выделяет отдельный пул потоков.
```

Bulkhead защищает от того, чтобы один медленный downstream не съел все
thread'ы приложения. Например, 100 потоков для платежей, 100 для email, 100
для аналитики — если email повис, платежи продолжают работать.

**Rate Limiter** — ограничение собственного RPS к downstream'у:

```java
@RateLimiter(name = "external-api", fallbackMethod = "fallback")
public ApiResponse call() { ... }
```

`limitForPeriod: 100, limitRefreshPeriod: 1s` → 100 RPS максимум.

**TimeLimiter** — таймаут на async вызов:

```java
@TimeLimiter(name = "api", fallbackMethod = "fallback")
public CompletableFuture<X> call() {
    return CompletableFuture.supplyAsync(() -> slow());
}
```

`timeoutDuration: 2s` → если CF не complete за 2с → TimeoutException, отменить.

**Композиция через AOP-аннотации:**

```java
@CircuitBreaker(name = "x")    // самый внешний
@RateLimiter(name = "x")
@Retry(name = "x")
@TimeLimiter(name = "x")
@Bulkhead(name = "x")          // самый внутренний
public CompletableFuture<X> call() { ... }
```

Порядок важен. Resilience4j по дефолту: `CircuitBreaker > RateLimiter > Retry
> TimeLimiter > Bulkhead`. Изменяется через `aspect-order`.

## Типичный вопрос на собесе

**Вопрос 1 (Лига, T-Bank, Альфа, Озон).** «Что такое Circuit Breaker и
зачем нужен?»

Паттерн отказоустойчивости. Защищает от каскадных отказов: если внешний
сервис тормозит/падает, CB размыкается и быстро отвечает ошибкой
(fail-fast), не давая нашему сервису висеть в ожидании. Три состояния:
CLOSED, OPEN, HALF_OPEN.

Без CB: timeout 30 сек × N RPS = thread pool starvation = весь сервис висит.
С CB: 1ms fail-fast при OPEN, downstream восстанавливается, проба через
HALF_OPEN.

**Вопрос 2 (T-Bank).** «Расскажи состояния CB.»

CLOSED — норма, считаем failure rate. OPEN — все calls fail-fast, ждём
recovery period. HALF_OPEN — пробуем N запросов; success → CLOSED, fail →
обратно в OPEN.

**Вопрос 3 (Альфа).** «Какие проблемы у retry?»

1. **Retry storm**: 1000 клиентов retry-ят одновременно → выводят
   восстанавливающийся сервис из строя снова. Лекарство: jitter (random
   delay).
2. **Бесконечный retry**: thread/pool starvation. Лекарство: maxAttempts.
3. **Retry на 4xx**: бесполезно, ошибка не transient. Лекарство:
   retryOn-список конкретных исключений.
4. **Retry на POST без idempotency**: дубликаты. Лекарство:
   Idempotency-Key.

**Вопрос 4 (Сбер).** «Чем Hystrix отличается от Resilience4j?»

Hystrix — Netflix, deprecated с 2018. Был тяжёлый (отдельный thread pool на
каждый command), сложно тестировать. Resilience4j — modern (Resilience4j
team), lightweight, без зависимости от RxJava, composable через builder и
аннотации, functional API. Default выбор для новых проектов с 2020+.

## Один день из практики

Тебя позвали: «integration с PSP (платёжной системой) висит. При сбое нашего
PSP — наш checkout endpoint умирает». Открываем код:

```java
@Service
public class CheckoutService {
    private final PspClient client;   // HTTP client с 30s timeout

    public CheckoutResult checkout(Cart cart) {
        try {
            PaymentResult pay = client.charge(cart);   // блокирующий HTTP
            return CheckoutResult.success(pay);
        } catch (Exception e) {
            return CheckoutResult.failure(e.getMessage());
        }
    }
}
```

При сбое PSP: 30 сек на каждый checkout. 100 RPS → 3000 thread'ов висят. Tomcat
pool 200 — исчерпан. Все остальные endpoint'ы виснут.

Шаг 1. Добавляем Resilience4j stack:

```java
@Service
public class CheckoutService {
    private final PspClient client;

    @CircuitBreaker(name = "psp", fallbackMethod = "checkoutFallback")
    @Retry(name = "psp")
    @TimeLimiter(name = "psp")
    public CompletableFuture<CheckoutResult> checkout(Cart cart) {
        return CompletableFuture.supplyAsync(() -> {
            PaymentResult pay = client.charge(cart);
            return CheckoutResult.success(pay);
        });
    }

    public CompletableFuture<CheckoutResult> checkoutFallback(Cart cart, Throwable t) {
        log.warn("PSP unavailable", t);
        // ставим в очередь outbox для retry потом:
        outboxService.queueRetry(cart);
        return CompletableFuture.completedFuture(
            CheckoutResult.deferred("Payment queued, will retry"));
    }
}
```

```yaml
resilience4j:
  circuitbreaker:
    instances:
      psp:
        slidingWindowSize: 50
        failureRateThreshold: 30
        slowCallRateThreshold: 60
        slowCallDurationThreshold: 1500ms
        waitDurationInOpenState: 60s
        permittedNumberOfCallsInHalfOpenState: 5
        minimumNumberOfCalls: 10
  retry:
    instances:
      psp:
        maxAttempts: 2
        waitDuration: 200ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
        retryExceptions:
          - java.net.ConnectException
          - java.io.IOException
        ignoreExceptions:
          - com.acme.PaymentDeclinedException   # бизнес-ошибка, не retry
  timelimiter:
    instances:
      psp:
        timeoutDuration: 3s
        cancelRunningFuture: true
```

Симуляция сбоя PSP: убиваем psp container. Через 10 fail-calls CB перешёл в
OPEN. Все следующие checkout'ы получают fallback за < 5мс. Сервис жив.

Через 60 сек CB пробует HALF_OPEN, 5 calls делает к PSP. PSP всё ещё down —
обратно в OPEN. Восстанавливаем PSP — следующая HALF_OPEN успешна → CLOSED,
нормальная работа.

## Типичные провалы

Кейс 1. Где баг (retry storm)?

```java
@Retry(name = "external", maxAttempts = 3, waitDuration = 100ms)
public X call() { ... }
// jitter не настроен
```

1000 клиентов одновременно retry-ят через ровно 100ms. Восстанавливающийся
downstream падает снова. Лекарство:

```yaml
retry:
  instances:
    external:
      maxAttempts: 3
      waitDuration: 100ms
      enableExponentialBackoff: true
      randomizedWaitFactor: 0.5   # jitter ±50%
```

Кейс 2. Где баг (retry на POST)?

```java
@Retry(name = "psp")
public PaymentResult charge(PaymentRequest req) { ... }
```

POST без idempotency-key → retry двойного списания. Лекарство: передавать
`Idempotency-Key: <uuid>` в HTTP header, PSP проверяет дубликаты по нему.
Либо: retry только на ConnectException (не дошло) — не на TimeoutException
(могло дойти).

Кейс 3. Где баг (CB на critical path)?

```java
@CircuitBreaker(name = "user", fallbackMethod = "fallback")
public User getById(Long id) { ... }

private User fallback(Long id, Throwable t) {
    return null;     // ✗
}
```

Fallback возвращает null. Caller получает NPE. Лекарство: либо exception,
либо cached version, либо `Optional.empty()`.

Кейс 4. Где баг (один CB на всё)?

```java
@CircuitBreaker(name = "global")
public X paymentCall() { ... }

@CircuitBreaker(name = "global")
public Y notificationCall() { ... }
```

Сбой email-сервиса размыкает CB → платежи тоже отключаются. Лекарство:
отдельный CB на каждый downstream.

Кейс 5. Где баг?

```java
@CircuitBreaker(name = "x")
public X call() {
    try {
        return riskyCall();
    } catch (Exception e) {
        return X.empty();   // ✗ exception "проглочен" — CB не видит failure
    }
}
```

CB меряет failure rate по выброшенным exception'ам. Если catch'им — CB не
учитывает. Лекарство: rethrow или объявить `recordFailurePredicate`.

Кейс 6. Где баг (без TimeLimiter)?

```java
@CircuitBreaker(name = "x")
public X call() {
    return slow30sCall();    // CB не имеет встроенного timeout
}
```

CB сам по себе не таймаутит. 30-сек запрос — 30 сек ожидания. Лекарство:
`@TimeLimiter` + async вызов, либо HTTP-клиент с явным timeout.

Кейс 7. Где баг (Bulkhead для blocking IO)?

```java
@Bulkhead(name = "x", type = THREADPOOL, maxThreadPoolSize = 100)
public X call() { ... }
```

100 потоков для одного downstream'а — для high RPS может быть мало или много.
Калибровать на нагрузочных тестах.
