## Зачем

Propagation определяет поведение, если уже открыта транзакция: (1) REQUIRED (default) — присоединиться к существующей или открыть новую; (2) REQUIRES_NEW — приостановить текущую, открыть новую (физически отдельный коннект! commit/rollback независимо); (3) NESTED — savepoint в текущей транзакции (commit при выходе из метода, rollback откатывает только до savepoint); (4) SUPPORTS — присоединиться, если есть, иначе без транзакции; (5) MANDATORY — обязательна внешняя; (6) NEVER — не должно быть внешней; (7) NOT_SUPPORTED — приостановить и работать без; (8) NESTED. REQUIRES_NEW — самое тяжёлое: лишний коннект, может deadlock'нуться с внешней. NESTED работает только на JDBC-savepoints (не везде).

## Ключевые различения

- REQUIRED (присоединиться/новая) ↔ REQUIRES_NEW (новая физическая)
- NESTED (savepoint) ↔ REQUIRES_NEW (отдельный коннект)
- MANDATORY (требует внешнюю) ↔ NEVER (запрещает внешнюю)
- SUPPORTS (без транзакции, если нет) ↔ NOT_SUPPORTED (suspend + без)

## Технология

- @Transactional(propagation = Propagation.REQUIRES_NEW)
- @Transactional(propagation = Propagation.NESTED)
- JpaTransactionManager / DataSourceTransactionManager
- JpaProperties.setOpenInView (антипаттерн!)

## Практика (упражнения)

- Реализовать @Transactional(REQUIRES_NEW) для логирования действий — даже при rollback внешней транзакции логи сохраняются
- NESTED для частичного rollback (savepoint)
- Спровоцировать deadlock через REQUIRES_NEW на одной строке таблицы

## Маркеры освоения

- Знаешь, что REQUIRES_NEW требует второй коннект
- Понимаешь, когда NESTED работает, а когда нет
- Не используешь REQUIRES_NEW «на всякий случай»

## Антипаттерны

- REQUIRES_NEW в цикле — каждый раз новый коннект, исчерпание пула
- Полагаться на NESTED без проверки, что JDBC-driver поддерживает savepoints
- Использовать default REQUIRED для всего — иногда нужна изоляция

## Развёрнутое объяснение

Propagation определяет поведение `@Transactional`-метода, если уже **открыта**
внешняя транзакция. Семь значений (Spring `Propagation` enum):

```
REQUIRED       (default) — присоединиться к существующей или открыть новую.
REQUIRES_NEW   — приостановить текущую, открыть НОВУЮ физическую транзакцию (новый коннект!).
NESTED         — savepoint в текущей транзакции; rollback откатывает только до savepoint.
SUPPORTS       — присоединиться, если есть; иначе — без транзакции.
NOT_SUPPORTED  — приостановить текущую, выполнить БЕЗ транзакции.
MANDATORY      — требует существующую транзакцию; иначе — IllegalTransactionStateException.
NEVER          — запрещает существующую; иначе — exception.
```

**REQUIRED — default.** 95% случаев. «Join existing or start new». Все вложенные
@Transactional методы участвуют в одной TX. Один rollback в любом месте
→ rollback всей.

```java
@Transactional   // REQUIRED по дефолту
public void outer() {
    repo.saveA();
    inner();      // присоединяется к outer TX
}

@Transactional
public void inner() {
    repo.saveB();
    if (badCondition) throw new RuntimeException();  // rollback всей TX
}
```

**REQUIRES_NEW — физически новая транзакция.** Spring suspend'ит текущую,
просит у `PlatformTransactionManager` НОВЫЙ Connection из пула, открывает на нём
новую TX. После метода — commit/rollback **только** новой. Outer
возобновляется. Это значит **два коннекта одновременно занято** этим thread'ом.

```java
@Transactional   // outer
public void mainFlow() {
    repo.save(order);
    audit.log(orderId);          // REQUIRES_NEW
    if (badCondition) throw new RuntimeException();
    // outer rollback, но audit уже закоммитилась!
}

@Service
public class Audit {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(Long orderId) { ... }
}
```

Применение: audit log, метрики, financial operations, которые ДОЛЖНЫ сохраниться
независимо от outer'а.

**Опасности REQUIRES_NEW:**

1. **Pool exhaustion в цикле:**
   ```java
   @Transactional
   public void processAll(List<X> xs) {
       for (X x : xs) {
           audit.log(x);   // REQUIRES_NEW × N
       }
   }
   ```
   Каждый итер занимает второй коннект. Hikari pool 20 → если 10
   processAll одновременно, 20 коннектов выбраны. Pool starvation.

2. **Deadlock с outer:**
   ```java
   @Transactional
   public void update(Long id) {
       repo.lockRow(id);       // SELECT FOR UPDATE на row id
       inner(id);              // REQUIRES_NEW
   }
   @Transactional(propagation = Propagation.REQUIRES_NEW)
   public void inner(Long id) {
       repo.lockRow(id);       // ждёт outer, который ждёт inner
   }
   ```
   Outer держит row lock, inner просит тот же lock на новом коннекте.
   Deadlock. БД через `lock_timeout` обнаружит.

**NESTED — savepoint.** Внутри outer'а создаётся savepoint (SQL: `SAVEPOINT
sp1`). Rollback вложенного откатывает только до savepoint, outer продолжает.
Требует поддержки JDBC-savepoints (PostgreSQL, MySQL InnoDB, Oracle, MSSQL — все
поддерживают; SQLite — нет).

```java
@Transactional   // REQUIRED
public void main() {
    repo.saveA();
    try {
        nested();   // NESTED — savepoint
    } catch (Exception e) {
        // outer продолжает, B откатилось, A осталось
    }
    repo.saveC();
}

@Transactional(propagation = Propagation.NESTED)
public void nested() {
    repo.saveB();
    throw new RuntimeException();  // rollback до savepoint
}
```

Применение: «попробуй сделать B, если не получится — не страшно, продолжай
A→C». Менее затратно, чем REQUIRES_NEW (один коннект, без suspend).

**SUPPORTS / NOT_SUPPORTED / MANDATORY / NEVER** — нишевые.

- `SUPPORTS` — «если кто-то открыл TX, используй её; иначе работаем без
  TX». Применение: для read-методов, которые могут участвовать в outer TX
  для consistency, а самостоятельно могут работать в auto-commit.
- `NOT_SUPPORTED` — «приостанови outer и работай без TX». Применение: тяжёлые
  reports/долгие запросы, которые не должны держать TX.
- `MANDATORY` — «обязана быть outer». Применение: метод, который должен
  быть частью большей операции (например, в `Repository` низкого уровня).
- `NEVER` — «outer запрещён». Очень редкое применение.

**Isolation levels.** Параметр `isolation`:

```
DEFAULT          — default БД (PostgreSQL: READ COMMITTED, MySQL: REPEATABLE READ)
READ_UNCOMMITTED — допускает dirty read
READ_COMMITTED   — нет dirty read; non-repeatable read возможен
REPEATABLE_READ  — нет dirty + non-repeatable; phantom read возможен
SERIALIZABLE     — нет ничего
```

См. `db-acid-isolation` для глубокого разбора.

**timeout и rollback.** Параметр `timeout` (секунды) задаёт максимальное время
TX. Если превышено — Spring бросит `TransactionTimedOutException`. Полезно для
страховки от длинных запросов.

```java
@Transactional(timeout = 5)
public void quickWork() { ... }
```

## Типичный вопрос на собесе

**Вопрос 1 (Лига, T-Bank, Альфа, VK).** «Какие propagation в Spring?»

Семь: REQUIRED (default), REQUIRES_NEW, NESTED, SUPPORTS, NOT_SUPPORTED,
MANDATORY, NEVER. Главные — REQUIRED, REQUIRES_NEW, NESTED.

Доводка: «когда REQUIRES_NEW?» — audit log, метрики, операции, которые должны
сохраниться независимо от outer rollback.

**Вопрос 2 (T-Bank, Сбер).** «Чем REQUIRES_NEW отличается от NESTED?»

REQUIRES_NEW — **новая физическая** транзакция, новый Connection из пула,
outer suspend'ится. Commit/rollback независимо.

NESTED — savepoint **в той же** транзакции. Один коннект. Rollback откатывает
до savepoint, outer продолжает. Дешевле, но требует JDBC savepoint support.

**Вопрос 3 (Альфа, Озон).** «Что не так?»

```java
@Transactional
public void process(List<Item> items) {
    for (Item i : items) {
        processOne(i);
    }
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void processOne(Item i) {
    repo.save(i);
}
```

(1) Self-invocation — `this.processOne` мимо прокси. REQUIRES_NEW не сработает.

(2) Даже если бы сработало — REQUIRES_NEW × N → каждая итерация занимает
второй коннект. Pool exhaustion.

Лекарство:
- Разнести в два сервиса.
- Использовать REQUIRED вместо REQUIRES_NEW, если изоляция не нужна.

**Вопрос 4 (VK).** «Будет ли rollback всей outer-транзакции, если inner с
REQUIRES_NEW бросит?»

Inner с REQUIRES_NEW — отдельная физическая TX. Её rollback откатывает только
inner. Outer продолжается. Если inner-exception долетит до outer — outer
тоже может попасть в rollback (по default rollback semantics).

```java
@Transactional   // REQUIRED
void outer() {
    repo.saveA();
    try {
        innerService.foo();   // REQUIRES_NEW, бросит RuntimeException
    } catch (RuntimeException e) {
        // outer продолжит, A сохранится, inner откатилось
    }
}
```

## Один день из практики

Тебя позвали разобраться: «отчёт по ежедневным транзакциям иногда содержит
дубликаты, иногда — пропущенные записи. Не понимаем». Открываем код:

```java
@Service
public class DailyReportService {
    @Transactional
    public void generate(LocalDate day) {
        List<Transaction> txs = txRepo.findByDay(day);
        for (Transaction tx : txs) {
            try {
                processOne(tx);     // self-invocation!
            } catch (Exception e) {
                log.warn("Failed: {}", tx.id(), e);
            }
        }
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    private void processOne(Transaction tx) {
        ReportEntry entry = compute(tx);
        reportRepo.save(entry);
        if (entry.amount().compareTo(SUSPICION_THRESHOLD) > 0) {
            throw new SuspiciousException("Manual review needed");
        }
    }
}
```

Проблемы:
1. `processOne` — `private` + self-invocation. AOP не работает. Метод вызывается
   как обычный Java-call, без транзакции, без REQUIRES_NEW.
2. Внутри outer-TX каждый save идёт в outer (REQUIRED-родная TX).
3. На SuspiciousException default rollback rule (RuntimeException → rollback) —
   откатит outer-TX. Все ранее сохранённые ReportEntry в этой outer тоже откатятся.
4. Catch внутри for ловит exception, но flag rollback-only уже выставлен — outer
   при выходе из generate всё равно сделает rollback.

Симптом «иногда дубликаты, иногда пропуски» — потому что после неудачной
suspicious entry transaction отменяется, затем при повторном запуске часть
данных уже есть, часть нет.

Фикс:

```java
@Service
@RequiredArgsConstructor
public class DailyReportService {
    private final TxProcessor txProcessor;
    private final TransactionRepository txRepo;

    public void generate(LocalDate day) {
        List<Transaction> txs = txRepo.findByDay(day);
        for (Transaction tx : txs) {
            try {
                txProcessor.processOne(tx);   // через прокси, в своей REQUIRES_NEW
            } catch (Exception e) {
                log.warn("Failed: {}", tx.id(), e);
            }
        }
    }
}

@Service
@RequiredArgsConstructor
public class TxProcessor {
    private final ReportRepository reportRepo;

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void processOne(Transaction tx) {        // public!
        ReportEntry entry = compute(tx);
        reportRepo.save(entry);
        if (entry.amount().compareTo(SUSPICION_THRESHOLD) > 0) {
            throw new SuspiciousException("...");
        }
    }
}
```

Теперь:
- `generate` — без `@Transactional` (отчёт — не атомарная операция).
- `processOne` — public в отдельном сервисе. REQUIRES_NEW работает.
- На SuspiciousException откатывается только эта одна entry, остальные сохраняются.
- Catch в generate просто логирует.

Регресс: после релиза дубликаты ушли, пропуски ушли. Урок: REQUIRES_NEW
работает только на public + external call.

## Типичные провалы

Кейс 1. Где баг?

```java
@Transactional
public void main() {
    repo.save(a);
    audit.log(...);    // REQUIRES_NEW — но self-invocation
    throw new RuntimeException();
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
private void log(...) { ... }
```

`private` + self-invocation. REQUIRES_NEW не срабатывает. Audit log в той же
outer TX → rollback вместе с main.

Кейс 2. Где баг (REQUIRES_NEW × N)?

```java
@Service
public class Worker {
    @Transactional
    public void process(List<X> xs) {
        for (X x : xs) {
            otherService.workOn(x);   // REQUIRES_NEW
        }
    }
}
```

Каждая итерация — новый Connection. На 1000 items × 50 параллельных process =
50K connections в пуле, exhaustion. Лекарство: REQUIRED или batch operations.

Кейс 3. Где баг (deadlock REQUIRES_NEW)?

```java
@Transactional
public void update(Long id) {
    repo.lockRow(id);     // SELECT FOR UPDATE
    auditService.log(id); // REQUIRES_NEW, который тоже SELECT FOR UPDATE на id
}
```

Outer держит lock на row id. Inner ждёт тот же row. Outer держит inner.
Deadlock в БД, обнаруживается через `lock_timeout`. Лекарство: audit в
отдельную таблицу (без overlap по id) или другую propagation.

Кейс 4. Где баг (NESTED без savepoint support)?

```java
@Transactional(propagation = Propagation.NESTED)
public void inner() { ... }

// JDBC connection с auto-commit, без savepoint support
```

`DataSourceTransactionManager` поддерживает savepoint только если Connection
поддерживает (`conn.getMetaData().supportsSavepoints()`). На SQLite, на
некоторых ORM-обёртках savepoint не работает → Spring бросит
`NestedTransactionNotSupportedException`. Лекарство: проверить driver.

Кейс 5. Где баг?

```java
@Transactional(propagation = Propagation.MANDATORY)
public void requireOuter() { ... }

// Вызывается напрямую без outer TX
service.requireOuter();   // IllegalTransactionStateException
```

MANDATORY требует существующей TX. Если caller не открыл — exception.
Полезно для low-level repository, который не должен открывать TX сам.

Кейс 6. Где баг (Hibernate flush)?

```java
@Transactional
public void main() {
    User u = userRepo.findById(1L);
    u.setName("New");          // dirty checking запомнило
    inner();                   // REQUIRES_NEW
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void inner() {
    User check = userRepo.findById(1L);   // видит ли изменение?
    // нет: outer TX suspend'ится, change не flush'ится
}
```

При suspend Hibernate не делает flush. Inner видит «старое» значение в БД.
Если ожидался snapshot после change — баг. Лекарство: `entityManager.flush()`
перед REQUIRES_NEW.

Кейс 7. Где баг (программный TX)?

```java
@Transactional
public void main() {
    transactionTemplate.execute(status -> {     // вложенная программная TX
        repo.save(...);
        return null;
    });
}
```

`TransactionTemplate` по умолчанию использует REQUIRED — он присоединяется к
существующей. Если нужна изоляция — `transactionTemplate.setPropagationBehavior(
TransactionDefinition.PROPAGATION_REQUIRES_NEW)`.
