## Зачем

volatile поле гарантирует: (1) видимость — write публикуется во main memory сразу, read берёт из main memory; (2) запрет reordering — компилятор/JIT не переставляет операции вокруг volatile. Но volatile НЕ атомарен. counter++ — это 3 операции: read → +1 → write; между read и write другой поток может вмешаться. Для атомарных операций нужны AtomicInteger.incrementAndGet (CAS-based) или synchronized. Применение volatile: status-флаги, double-checked locking singleton, публикация ссылки на immutable объект. Memory barrier — load на read, store на write. На x86 read volatile почти бесплатен, write — дороже из-за store barrier.

## Ключевые различения

- Видимость (publish) ↔ Атомарность (read-modify-write)
- volatile (видимость + ordering) ↔ AtomicInteger (видимость + CAS-атомарность)
- volatile (single var) ↔ synchronized (compound action)
- Java 5 memory model — гарантии volatile, ReentrantLock, final

## Технология

- volatile
- AtomicInteger / AtomicLong / AtomicReference (CAS)
- LongAdder (для high-contention counters)
- VarHandle (Java 9+) — низкоуровневые операции

## Практика (упражнения)

- Запустить 100 потоков × 10000 increment volatile int — увидеть, что итог != 1_000_000
- Заменить на AtomicInteger.incrementAndGet — итог == 1_000_000
- Реализовать DCL Singleton с volatile

## Развёрнутое объяснение

`volatile` — самое тонкое ключевое слово в Java. На junior его объясняют как
«публикует в main memory», на middle — как memory barrier + ordering, на senior —
как часть JMM с конкретной семантикой `synchronizes-with` (см. JLS §17.4).

**Что гарантирует volatile (Java 5+, JSR-133):**

1. **Видимость (visibility)**: write volatile-поля **happens-before** read того же
   поля из другого потока. Все изменения, видимые писателю до write, становятся
   видимыми читателю после read.
2. **Запрет reordering**: компилятор и JIT не могут переставлять операции через
   volatile-чтение/запись. Конкретно: операции **до** volatile-write не могут
   переехать **после** неё; операции **после** volatile-read не могут переехать
   **до** неё.
3. **Атомарность для long/double**: volatile-чтение/запись 64-битного поля
   атомарны (без volatile они могут быть split на 2×32-битных операции на
   32-битных JVM, см. JLS §17.7).

**Чего volatile НЕ гарантирует:**

- **Атомарность read-modify-write**. `counter++` = `read` → `+1` → `write`. Между
  read и write другой поток может вмешаться. Это самая частая ошибка на собесе.

```java
private volatile int counter = 0;

// 100 потоков × 10 000 increment
for (int i = 0; i < 10000; i++) counter++;

// Итог: ≈ 600 000 - 800 000, не миллион. Потеря инкрементов.
```

**Memory barrier на уровне CPU.** В JLS volatile определён через
happens-before, но реализуется через barrier'ы. JIT генерирует:
- **Перед volatile-write**: StoreStore barrier (нельзя переехать предыдущим write через эту строку) + LoadStore.
- **После volatile-write**: StoreLoad barrier (самый дорогой, синхронизирует store-буфер с памятью).
- **Перед volatile-read**: LoadLoad + LoadStore barrier.

На x86 (с TSO модель консистентности) volatile-read почти бесплатен (только
запрет компиляторного reordering), volatile-write требует `mfence` или `xchg`
для StoreLoad. На ARM (weak memory model) barrier'ы дороже — `dmb ish` инструкция.

**Применения volatile, которые работают:**

```java
// 1. Stop-флаг
private volatile boolean stopRequested;

public void start() {
    new Thread(() -> {
        while (!stopRequested) {        // volatile read
            doWork();
        }
    }).start();
}

public void stop() { stopRequested = true; }   // volatile write

// 2. Double-checked locking singleton (volatile обязателен — иначе reordering)
private static volatile Heavy instance;

public static Heavy get() {
    Heavy h = instance;
    if (h == null) {
        synchronized (Heavy.class) {
            h = instance;
            if (h == null) {
                instance = h = new Heavy();    // volatile write публикует объект
            }
        }
    }
    return h;
}

// 3. Публикация immutable снапшота
private volatile Map<String, Config> configs = new HashMap<>();

public void reload() {
    Map<String, Config> next = loadFromDb();    // строим новый
    configs = next;                              // атомарный swap, видим всем
}
```

**Применения, где volatile НЕ хватает:**

```java
// 1. Счётчики — используйте AtomicInteger или LongAdder
// 2. Compound action (containsKey + put) — synchronized или CHM
// 3. Сохранение инвариантов между полями:
private volatile int x, y;   // OK по отдельности, но (x, y) не consistent snapshot
```

**AtomicXxx изнутри.** `AtomicInteger.incrementAndGet` использует CAS-цикл:

```java
public final int incrementAndGet() {
    int current, next;
    do {
        current = get();         // volatile read
        next = current + 1;
    } while (!compareAndSet(current, next));   // CAS
    return next;
}
```

CAS = атомарная инструкция CPU (`lock cmpxchg` на x86, `ldxr/stxr` пара на ARM).
Если concurrent поток успел изменить — CAS возвращает false, retry. В Java 8+
есть упрощённый `incrementAndGet` через `Unsafe.getAndAddInt` (intrinsic, JIT
заменяет на native).

**LongAdder и Striped64.** При high contention CAS-loop на AtomicLong становится
bottleneck — все потоки бьются за одну volatile-ячейку, cache line ping-pong
между ядрами. Решение Doug Lea — `LongAdder`: внутри массив `Cell[]` (padded
по cache line, 128 байт каждая), каждый поток пишет в свою ячейку через `threadId %
cells.length`. `sum()` агрегирует. На 32+ потоках LongAdder x5-x10 быстрее
AtomicLong:

```java
LongAdder counter = new LongAdder();
counter.increment();   // O(1) без contention
counter.sum();         // O(cells), но редкий
```

## Типичный вопрос на собесе

**Вопрос 1 (Лига, T-Bank, Альфа, VK, Сбер, Озон).** «Что делает volatile? Чем
отличается от synchronized?»

Ответ:
1. **volatile** гарантирует видимость (write happens-before read того же поля) и
   запрет reordering для одной переменной. НЕ даёт атомарности RMW.
2. **synchronized** — взаимное исключение (один поток в критической секции) +
   видимость + reordering. Можно использовать для compound actions.
3. **volatile** дешевле (no lock, no monitor inflation), но решает только узкий
   класс задач — флаги, публикация immutable объектов, DCL.
4. **synchronized** — для compound action, RMW, сложных инвариантов.

Доводка: «volatile counter работает?» — нет. `counter++` не атомарен. Решение:
`AtomicInteger.incrementAndGet` или `LongAdder.increment`.

**Вопрос 2 (Альфа, Озон).** «Что не так?»

```java
public class Stats {
    private volatile int requests = 0;

    public void recordRequest() {
        requests++;     // ✗
    }

    public int getRequests() { return requests; }
}
```

`requests++` — read-modify-write, не атомарно. Под 100 потоков RPS итог теряет
30-50% инкрементов. Замена: `private final AtomicInteger requests = new AtomicInteger();`
+ `requests.incrementAndGet()`, либо `LongAdder` если read редкий.

**Вопрос 3 (T-Bank, продвинутый).** «Объясни, почему DCL singleton нужно volatile.»

```java
class Singleton {
    private static Singleton instance;     // БЕЗ volatile
    public static Singleton get() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();  // (*)
                }
            }
        }
        return instance;
    }
}
```

Проблема: строчка `instance = new Singleton()` на самом деле компилируется в три
шага:
1. Allocate memory.
2. Constructor run.
3. Assign reference to `instance`.

Без volatile JIT/CPU могут переставить (3) перед (2). Тогда другой поток,
читающий `instance` без synchronized, может увидеть **partially constructed**
объект — ссылка не null, но конструктор ещё не отработал. NPE/случайное
поведение.

`volatile` запрещает reordering write через себя. (3) с volatile-write
происходит ПОСЛЕ (2). Другой поток через volatile-read получит fully constructed
объект.

Альтернатива — Holder idiom (без volatile, без synchronized):

```java
class Singleton {
    private static class Holder { static final Singleton INSTANCE = new Singleton(); }
    public static Singleton get() { return Holder.INSTANCE; }
}
```

JLS гарантирует thread-safe init статических полей класса.

**Вопрос 4 (Сбер).** «Чем `volatile long` отличается от обычного `long` на
32-битной JVM?»

JLS §17.7: чтение/запись 64-битного поля (long/double) без volatile НЕ гарантирует
атомарность на 32-битных JVM — split на 2×32 операции, между которыми другой
поток может увидеть «половинку». volatile делает их атомарными. Для int/short/byte/
references на 64-bit JVM это default; на современных серверах редко актуально,
но в embedded/Android до 2018 встречалось.

## Один день из практики

Тебе пришёл алёрт: «счётчик запросов в Prometheus не совпадает с access-log».
Лог говорит 1.2M за час, метрика — 750K. Что отбросилось?

Открываем metric collector:

```java
@Component
public class MetricsCollector {
    private volatile long requests = 0;
    public void incRequests() { requests++; }
    public long getRequests() { return requests; }
}
```

`requests++` под нагрузкой теряет 30-50% инкрементов. volatile визуально создаёт
ощущение «надёжно», но это про **видимость**, а не атомарность.

Фикс 1 — AtomicLong:

```java
private final AtomicLong requests = new AtomicLong();
public void incRequests() { requests.incrementAndGet(); }
public long getRequests() { return requests.get(); }
```

Но под 10K RPS на 32 ядрах CAS-loop сам по себе становится bottleneck.
JFR показывает ~12% CPU на CAS-цикл.

Фикс 2 — LongAdder:

```java
private final LongAdder requests = new LongAdder();
public void incRequests() { requests.increment(); }
public long getRequests() { return requests.sum(); }   // O(cells)
```

После замены счётчик совпадает с access-log, CPU upon counter — 0.3%. Урок:
- volatile = видимость, не атомарность.
- AtomicXxx = атомарность + видимость, но contention на одной cache line.
- LongAdder = striped counter, x10 throughput при high contention, mild
  inconsistency на sum() (приемлемо для metrics).

## Типичные провалы

Кейс 1. Где баг?

```java
private volatile int balance;
public void withdraw(int amount) {
    if (balance >= amount) {
        balance -= amount;
    }
}
```

`if (balance >= amount)` и `balance -= amount` — две операции, между ними
другой поток может тоже снять деньги. Получится отрицательный баланс.
Решение: `synchronized` или `AtomicInteger.updateAndGet(v -> v >= amount ? v - amount : v)`.

Кейс 2. Где баг?

```java
private volatile boolean ready;
private int data;

// Thread 1
data = compute();
ready = true;

// Thread 2
while (!ready) Thread.yield();
use(data);
```

Это РАБОТАЕТ! `data = compute()` happens-before `ready = true` (program order),
volatile-write happens-before volatile-read другого потока. Когда T2 видит
`ready == true`, он гарантированно видит и `data` (transitivity). Это
канонический пример: volatile-write **публикует** все изменения, видимые
писателю до неё. Эту идею ещё называют «volatile piggybacking».

Кейс 3. Где баг?

```java
private volatile List<Integer> list = new ArrayList<>();

public void add(Integer i) {
    list.add(i);    // мутация коллекции, не publishing
}
```

volatile на ссылке гарантирует видимость **самой ссылки**, но не операций над
объектом. `ArrayList.add` не thread-safe. Лекарство: `CopyOnWriteArrayList`,
или построить новый список + volatile swap:

```java
synchronized (this) {
    List<Integer> next = new ArrayList<>(list);
    next.add(i);
    list = next;     // volatile-publish
}
```

Кейс 4. Где баг?

```java
private final List<Long> data = new ArrayList<>();

// Thread 1 готовит данные
data.add(1L);
data.add(2L);
// «отдаёт» в Thread 2 через флаг
ready = true;

// Thread 2
while (!ready) Thread.yield();
processAll(data);
```

`ready` — обычное поле, не volatile. T2 может вечно вертеться (JIT держит
ready в регистре), или увидеть partially-built ArrayList. Лекарство:
`volatile boolean ready` или `CountDownLatch`.

Кейс 5. Где баг?

```java
private volatile UserConfig config;
public void update(UserConfig newConfig) {
    config.timezone = newConfig.timezone;    // мутация publish-immutable объекта
    config.locale = newConfig.locale;
}
```

UserConfig использовался как «публикуемый immutable snapshot», но кто-то
изменил mutable поля. Между двумя присваиваниями другой читатель видит
смешанное состояние (новый timezone + старый locale). Лекарство: сделать
UserConfig полностью immutable (final поля, без сеттеров) и `config = newConfig`
одним volatile-write.

Кейс 6. Где баг (DCL без volatile)?

```java
class Lazy {
    private static Heavy instance;          // должно быть volatile
    public static Heavy get() {
        if (instance == null) {
            synchronized (Lazy.class) {
                if (instance == null) {
                    instance = new Heavy();
                }
            }
        }
        return instance;
    }
}
```

См. вопрос 3 — без volatile другой поток может увидеть half-constructed object.

Кейс 7. Где баг (compound check-then-act)?

```java
private volatile boolean initialized;
public void initOnce() {
    if (!initialized) {
        doExpensiveInit();
        initialized = true;
    }
}
```

Между check и act другой поток тоже зайдёт в if (volatile-read оба true прочитают
false до изменения). doExpensiveInit выполнится N раз. Лекарство: synchronized
блок, или DCL pattern.
